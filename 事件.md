## 事件1: 启动服务端口占用循环

### 问题描述
在反复调试「安装向导状态检测」功能时，多次出现以下循环：
1. 修改 `installService.ts` 代码后需要重启后端服务
2. 旧进程仍占用 3001 端口
3. 新启动的服务报 `EADDRINUSE: address already in use`
4. 被迫使用 `taskkill` 终止进程
5. 整个过程反复进行，浪费大量时间

### 根本原因
- `tsx watch` 虽然支持热重载，但文件路径解析变更时 watch 可能失效
- 没有统一的「停止服务」脚本，每次手动查进程 ID

### 解决方案
1. **停止脚本**: 使用项目根目录的 `停止.bat`（如果存在），或统一通过 PowerShell 执行：
   ```
   netstat -ano | findstr ":3001" | findstr "LISTENING"
   taskkill /F /PID <进程ID>
   ```
2. **下次避免**: 修改服务端核心配置文件前，先停止服务再修改，减少重启次数

### 教训
> 修改涉及路径解析、配置目录定位的代码时，应先完成代码逻辑评审，再执行启动验证。不要边改边启，造成端口占用循环。

## 事件2: 安装向导配置文件路径不统一

### 问题描述
1. 前端检测 `GET /api/installer/setup-status` 返回 `{completed: false, hasConfig: false}`
2. 项目启动后页面跳转到安装向导，而不是直接进入看板
3. 调试发现 `installService.ts` 中的 `getConfigDir()` 存在路径解析问题
4. `__dirname` 在 tsx 上下文中解析为编译后的目录，而非项目根目录
5. 配置文件放在 `D:\AI-Workspace\ZhiXu-ACOP\data\` 中，但代码找不到它

### 根本原因
- `path.resolve(__dirname, '..', '..', '..')` 在 tsx watch 模式下解析路径不稳定
- 没有使用单一的配置路径策略
- 没有在启动日志中打印配置文件路径，导致调试困难

### 解决方案
1. **统一路径策略**: 强制使用 `process.cwd()` + `./data` 作为配置目录
2. **启动日志**: 在 `loadInstallConfig()` 中打印实际查找的路径
3. **备用判断**: 如果项目 `data/` 目录存在且有数据文件，视为已安装完成

### 修复的代码位置
- `src/server/services/installService.ts`
  - `getConfigDir()`: 改用 `process.cwd()` 定位项目根目录
  - `getSetupStatus()`: 增加 `hasDataFiles` 备用判断
  - `loadInstallConfig()`: 增加详细日志输出

### 教训
> 路径解析问题在 Windows + tsx watch 环境下极易踩坑。优先使用 `process.cwd()` 而不是 `__dirname` 的相对路径导航。任何文件读取失败时，应第一时间打印出「实际尝试读取的路径」，而不是猜。

## 事件3: 前端页面加载异常（安装向导不该出现时出现）

### 问题描述
用户启动项目后，发现：
1. 左侧导航栏底部出现「安装向导」入口（不该出现）
2. 打开页面后跳转到安装向导页面（实际已完成安装）
3. 其他页面（看板、事件列表等）可能出现数据加载异常

### 根本原因
1. **导航栏显示问题**: `Layout.tsx` 的 `navItems` 数组中包含了安装向导项，应该移除
2. **页面跳转问题**: 后端 API `/api/installer/setup-status` 返回 `completed: false`
3. **前端判断逻辑**: `App.tsx` 中 `InstallerGate` 组件检测到 `completed: false` 后强制跳转到安装向导

### 解决方案
1. **前端 `Layout.tsx`**: 从 `navItems` 中移除 `{ path: '/installer', label: '安装向导', icon: '🚀' }`
2. **后端 `installService.ts`**: 修复 `getSetupStatus()` 使其正确识别项目 data 目录中的配置文件
3. **双重保险**: 即使 `install-config.json` 不存在，如果 data 目录中有事件文件、告警文件等真实数据，也应返回 `completed: true`

### 教训
> 「安装状态」是一个影响全局路由的关键状态，必须保证判断逻辑稳健。不能只依赖单个配置文件是否存在，应该增加多重判断条件（数据目录存在、数据文件数量 > 0 等），避免用户看到错误的安装向导页面。
